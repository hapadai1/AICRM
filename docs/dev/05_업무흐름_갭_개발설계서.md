## 3. G-11. 고객 진행 단계 ★ 신규 핵심

### 3.1 문제

PDF 1페이지 표 전체가 **"거래 1건이 상담부터 출고까지 어느 단계에 있는가"** 를 나타내는데, 현행 시스템에는 이 축이 없다. 존재하는 상태값은 전부 다른 목적이다.

| 현행 상태 | 관리 주체 | 한계 |
|---|---|---|
| `customers.status` (PROSPECT/CONTRACTED/INACTIVE) | 시스템 자동 | 고객 분류일 뿐, 진행 위치가 아님 |
| `order_items.status` / `components.status` | 시스템 계산(입출고 이벤트 집계) | 계약 이전(상담·계약 단계)이 표현 불가. 구성품별로 흩어져 "이 고객이 지금 어디"를 한눈에 못 봄 |
| `contracts.status` | 시스템 자동 | 계약서 자체의 상태 |

그 결과 **① 담당자가 고객 상황을 파악하려면 여러 화면을 뒤져야 하고, ② 고객 연락 시점을 시스템이 알려주지 못한다.** ②가 곧 G-06(연락 누락)의 근본 원인이다.

### 3.2 설계 원칙

1. **사람이 관리하는 표시 레이어**로 둔다. 기존 시스템 계산 상태(`order_items.status` 등)를 **대체하거나 자동 연동하지 않는다.** 이중 관리로 인한 불일치·롤백 문제를 원천 차단하기 위함이다.
2. 단계 전진은 **전적으로 수동**. 다만 시스템 상태와 어긋나면 화면에 **힌트만** 띄운다(강제·자동 전진 없음).
   > 예: 진행 단계는 `발주 요청`인데 구성품이 이미 `가봉입고` → *"가봉 입고 처리된 품목이 있습니다. 단계를 옮기시겠어요?"*
3. **단계 변경이 고객 연락의 유일한 트리거**다(G-06). 발송은 확인창을 거친다.
4. 단위는 **거래(주문) 1건**. 고객이 여러 계약을 가지면 진행도 여러 개. 단 계약 이전에는 주문이 없으므로 `orderId`는 나중에 채워진다.

### 3.4 스키마

```prisma
/** 진행 단계 마스터 (시드 고정, 이름·템플릿 매핑은 관리자 조정 가능) */
model JourneyStage {
  id             String   @id @db.Uuid
  /** CUSTOM | RENTAL */
  trackType      String   @map("track_type") @db.VarChar(20)
  code           String   @map("code") @db.VarChar(30)
  name           String   @map("name") @db.VarChar(60)
  sequenceNo     Int      @map("sequence_no")
  /** 이 단계 진입 시 제안할 알림 템플릿 (없으면 발송 제안 없음) */
  templateId     String?  @map("template_id") @db.Uuid
  active         Boolean  @default(true) @map("active")
  createdAt      DateTime @default(now()) @map("created_at") @db.Timestamptz(6)
  updatedAt      DateTime @updatedAt @map("updated_at") @db.Timestamptz(6)

  template NotificationTemplate? @relation(fields: [templateId], references: [id])
  events   JourneyEvent[]

  @@unique([trackType, code])
  @@unique([trackType, sequenceNo])
  @@map("journey_stages")
}

/** 거래 1건의 진행 (계약 전에는 orderId 없음) */
model CustomerJourney {
  id               String    @id @db.Uuid
  customerId       String    @map("customer_id") @db.Uuid
  orderId          String?   @map("order_id") @db.Uuid
  trackType        String    @map("track_type") @db.VarChar(20)
  currentStageCode String    @map("current_stage_code") @db.VarChar(30)
  /** ACTIVE | COMPLETED | CANCELLED */
  status           String    @default("ACTIVE") @map("status") @db.VarChar(20)
  startedAt        DateTime  @map("started_at") @db.Timestamptz(6)
  completedAt      DateTime? @map("completed_at") @db.Timestamptz(6)
  rowVersion       Int       @default(0) @map("row_version")
  createdAt        DateTime  @default(now()) @map("created_at") @db.Timestamptz(6)
  updatedAt        DateTime  @updatedAt @map("updated_at") @db.Timestamptz(6)

  customer Customer       @relation(fields: [customerId], references: [id])
  order    Order?         @relation(fields: [orderId], references: [id])
  events   JourneyEvent[]

  @@index([customerId, startedAt(sort: Desc)])
  @@index([status, currentStageCode])
  @@map("customer_journeys")
}

/** 단계 변경 이력 + 그때 발송한 알림 연결 */
model JourneyEvent {
  id                    String   @id @db.Uuid
  journeyId             String   @map("journey_id") @db.Uuid
  stageId               String   @map("stage_id") @db.Uuid
  fromStageCode         String?  @map("from_stage_code") @db.VarChar(30)
  toStageCode           String   @map("to_stage_code") @db.VarChar(30)
  /** 되돌리기 시 필수 */
  reason                String?  @map("reason")
  notes                 String?  @map("notes")
  /** SENT | DEFERRED | SKIPPED | NONE — 이 변경 시점의 고객 연락 처리 결과 */
  notificationOutcome   String   @default("NONE") @map("notification_outcome") @db.VarChar(20)
  notificationHistoryId String?  @map("notification_history_id") @db.Uuid
  actorId               String   @map("actor_id") @db.Uuid
  changedAt             DateTime @map("changed_at") @db.Timestamptz(6)

  journey             CustomerJourney      @relation(fields: [journeyId], references: [id])
  stage               JourneyStage         @relation(fields: [stageId], references: [id])
  notificationHistory NotificationHistory? @relation(fields: [notificationHistoryId], references: [id])
  actor               User                 @relation("JourneyEventActor", fields: [actorId], references: [id])

  @@index([journeyId, changedAt(sort: Desc)])
  @@map("journey_events")
}
```

- 신규 권한 `JOURNEY_EDIT` 1개만 추가. 조회는 기존 `CUSTOMER_VIEW` 재사용.
- 전이 규칙: 전진은 **건너뛰기 허용**(현장에서 단계가 생략되는 경우가 있음), **후진은 `reason` 필수**(`production-status.ts`의 기존 규칙과 동일한 철학).

| GET | `/customers/:id/journeys` | `CUSTOMER_VIEW` | 고객의 진행 목록(단계·이력 포함) |
| POST | `/customers/:id/journeys` | `JOURNEY_EDIT` | 진행 시작 `{ trackType, orderId?, startStageCode? }` |
| GET | `/journeys/:id` | `CUSTOMER_VIEW` | 상세 + 이력 |
| **POST** | **`/journeys/:id/stage`** | `JOURNEY_EDIT` | **단계 변경 (핵심)** |
| POST | `/journeys/:id/complete` \| `/cancel` | `JOURNEY_EDIT` | 종료 |
| GET | `/journeys?stage=&trackType=&stalledDays=` | `CUSTOMER_VIEW` | 단계별 현황(칸반·정체 조회) |

**단계 변경 응답 계약** — 발송 확인창의 근거가 된다.

```jsonc
// POST /journeys/:id/stage
// req: { toStageCode, reason?, notes?, version }
{
  "data": {
    "journey": { "id": "...", "currentStageCode": "PRODUCT_RECEIVED", "version": 4 },
    "suggestedNotification": {           // 템플릿 매핑이 없으면 null
      "templateId": "...",
      "templateCode": "PRODUCT_RECEIVED_NOTICE",
      "channel": "ALIMTALK",
      "recipientPhone": "010-1234-5678",
      "renderedBody": "김철수님, 주문하신 정장이 입고되었습니다...",
      "variables": { "고객명": "김철수", "품목": "정장" },
      "triggerKey": "journey:{journeyId}:PRODUCT_RECEIVED"   // 멱등키
    }
  }
}
```

- **단계 변경과 발송은 분리된 요청이다.** 발송 실패가 단계 변경을 롤백하면 안 되고, 담당자가 문구를 보고 취소할 수 있어야 하기 때문.
- 발송은 기존 `POST /notifications/send` 를 그대로 사용(응답의 `triggerKey` 전달 → 기존 멱등 로직이 중복 발송을 구조적으로 차단).
- 발송 결과는 `POST /journeys/:id/events/:eventId/notification-outcome { outcome, notificationHistoryId? }` 로 회신해 이력에 봉합한다.

## 4. G-06. 고객 연락 — 단계 변경 연동 (v1.0 전면 재설계)

**v1.0 폐기 사항**: `notification_rules.autoSend` 기반 자동발송 엔진, `@nestjs/schedule` 스케줄러, 트리거 9종 이벤트 훅. → **"완전 자동은 힘들다"는 요건에 따라 전부 취소.**

**v2.0 방식**: 연락 시점은 **사람의 단계 변경**이 정의하고, 시스템은 **문구를 준비해 확인창을 띄우는 역할**만 한다.

```
담당자가 단계 변경 → 서버가 템플릿 매핑 조회 → 치환된 문구를 응답에 실어 보냄
                  → 화면이 확인창 표시 → [발송] 시에만 실제 전송(멱등키 적용)
                  → 결과를 진행 이력에 기록 ([나중에]는 대시보드 연락대기로)
```

**적용 지점 2곳** (동일한 확인창 컴포넌트를 공유)
1. **진행 단계 변경** (§3) — 연락 단계 5종
2. **수선 상태 변경** — `POST /repairs/:id/status-events` 응답에 동일 형태의 `suggestedNotification` 추가. PDF "수선 입고 → 고객연락" 대응. 대상 상태: `RECEIVED`(접수 확인), `CUSTOMER_NOTIFIED`(수선 완료 안내).

**기존 자산 재사용** — 신규 발송 로직 없음
- `notification_templates` / `renderTemplate()` / `POST /notifications/send` / `idempotency_keys` 멱등 / `notification_history` 전부 현행 그대로.
- `notification_rules` 테이블은 **`journey_stages.templateId` 매핑으로 역할이 대체**되므로 신규 사용 없이 유지만 한다(미사용 테이블. 물리삭제 금지 원칙에 따라 존치).

**관리자 화면**: `NotificationsPage`에 `단계별 문구 매핑` 탭 — 단계 × 템플릿 연결, 연락 없는 단계는 비움.

**테스트 (신규 4건)**: 수선 상태 변경 시 제안 생성/미대상 상태는 `null`, 벤더 실패 시 상태 변경은 유지, 문구 치환 정확성.

## 9. 미결 사항

1. **연락 단계 5종의 실제 문구** — 시드 템플릿은 초안으로 넣고, 알림톡 승인 신청 시 확정 문구로 교체한다. (G-10 선행 조건이기도 함)
2. **정체 기준 일수** — 진행 현황 보드에서 "정체"로 강조할 기준. 잠정 7일로 두고 운영 피드백으로 조정.
3. **기존 진행 중 거래의 초기 단계 부여** — 운영 전환 시 기존 주문에 진행 레코드를 만들어야 한다. `order_items.status` → 단계 코드 매핑 일괄 생성 스크립트를 S1에 포함할지, 신규 거래부터 적용할지 결정 필요.
