# AICRM 구현표준정의서 v1.0

- 작성일: 2026-07-21
- 목적: 화면·API 정의서(12장 공통 규약)와 데이터모델설계서(3·14장 규칙)를 코드 수준 표준으로 확정한다.

---

## 1. 백엔드 표준 (NestJS + Prisma)

### 1.1 모듈 구성
- 도메인당 1모듈: `controller / service / dto / (필요시) repository` 구성.
- 상태 변경은 일반 PATCH와 분리한 **액션 엔드포인트**(`POST /contracts/:id/confirm` 등)로 구현한다.
- 외부 연동(네이버·알림톡·PG)은 `adapters/` 하위 인터페이스 + 구현체로 격리한다.

### 1.2 Prisma 규칙
- 모델명 PascalCase, DB 매핑은 `@@map("snake_case 복수형")`, 컬럼 `@map("snake_case")`.
- PK는 `String @id @db.Uuid`, 애플리케이션에서 uuid v4 생성.
- 공통 컬럼: `createdAt/updatedAt(timestamptz)`, 경합 테이블에 `rowVersion Int @default(0)`.
- 상태값은 varchar + TypeScript enum(공용 `common/enums.ts`)으로 관리한다. DB enum은 쓰지 않는다(코드 추가 시 마이그레이션 부담).
- Prisma가 표현 못 하는 제약(렌탈 기간 EXCLUDE, partial unique)은 마이그레이션 SQL에 수기 추가:
  ```sql
  CREATE EXTENSION IF NOT EXISTS btree_gist;
  ALTER TABLE rental_allocations ADD CONSTRAINT rental_allocation_no_overlap
    EXCLUDE USING gist (rental_inventory_item_id WITH =,
      daterange(pickup_date, availability_end_date, '[]') WITH &&)
    WHERE (status <> 'CANCELLED');
  ```

### 1.3 API 계약
- 기본 경로 `/api/v1`. 응답은 항상 envelope:
  - 성공: `{ "data": ..., "meta": { "requestId", "timestamp" } }`
  - 목록: `{ "data": [...], "page": { number, size, totalElements, totalPages }, "meta": ... }`
  - 오류: `{ "error": { "code", "message", "fieldErrors?", "details?" }, "meta": ... }`
- 전역 `ResponseInterceptor` + `GlobalExceptionFilter`로 구현. 컨트롤러는 순수 데이터만 반환.
- 오류 코드는 `common/error-codes.ts` 단일 파일에 정의(화면·API 정의서 15.2 목록 그대로). `BusinessException(code, message, httpStatus, details?)` 하나로 throw.
- 페이지네이션: `page`(1-base), `size`(기본 30, 최대 100). 날짜 `YYYY-MM-DD`, 일시 ISO-8601, 금액 정수 원, 치수 decimal cm.

### 1.4 인증·권한
- `Authorization: Bearer <accessToken>`. refresh token은 DB 저장(해시) + 회전.
- 권한은 `@RequirePermission('CONTRACT_CONFIRM')` 데코레이터 + `PermissionsGuard`. 권한 코드는 화면·API 정의서 부록 B를 시드로 고정.
- 로그인 5회 연속 실패 시 계정 LOCKED(관리자 해제).

### 1.5 동시성·트랜잭션·멱등성
- 쓰기 API는 요청 body의 `version`과 DB `row_version` 비교 → 불일치 시 409 `*_VERSION_CONFLICT`. 성공 시 `row_version + 1`.
- 계약 확정(버전 확정+고객 전환+주문·품목 생성), 작업지시서 출력(스냅샷+버전+파일), 렌탈 배정/ID 변경은 `prisma.$transaction` 단일 트랜잭션.
- `Idempotency-Key` 헤더: 계약 확정, 작업지시서 출력, 메시지 발송에 적용. `idempotency_keys` 테이블에 최초 응답 저장 후 재요청 시 그대로 반환.

### 1.6 감사로그
- `AuditService.log({ userId, action, entityType, entityId, before, after, reason })`를 서비스 계층에서 명시 호출(주요 CUD·확정·상태변경·출력).
- 비밀번호·토큰은 before/after에서 제외. 감사로그 수정·삭제 API 없음.

### 1.7 삭제 정책
- 업무 데이터는 물리 삭제 금지 → 상태 변경(CANCELLED/INACTIVE/RETIRED). 기준정보는 `active=false`.
- Prisma FK는 전부 RESTRICT 기본.

## 2. 프론트엔드 표준 (React + AntD)

- 화면 1개 = `features/<도메인>/<화면ID>` 컴포넌트. 라우트 경로는 화면 흐름 기준(`/customers/:id`, `/contracts/:id` 등).
- 서버 통신은 `api/` 의 axios 인스턴스만 사용. 응답 envelope는 인터셉터에서 `data` 언랩, 오류는 `{code, message, fieldErrors}`로 정규화.
- 401 → refresh 재시도 → 실패 시 로그인 이동(현재 URL 보존). 409 버전 충돌 → 공용 충돌 다이얼로그(재조회/취소).
- Query Key 규칙: `[도메인, id?, 하위목록?]` 예) `['customers', id, 'contracts']`. 변경 성공 시 관련 키 invalidate(고객 상세 요약 재조회 규칙 포함).
- 권한: 로그인 응답의 permission 목록을 스토어에 보관, 메뉴·버튼 노출 제어(`<Can permission="...">`). 서버 검증과 이중 적용.
- 상태 배지는 텍스트+색상 병기(색상 단독 금지). 미저장 변경 시 이탈 경고.
- 태블릿 채촌: 공용 `NumericKeypad` 컴포넌트(0~9, 소수점, 지우기, 이전/다음, 완료), 필드 높이 48px 이상.

## 3. 명명·코드 스타일

| 대상 | 규칙 |
|---|---|
| TS 변수/함수 | camelCase, 클래스/타입 PascalCase |
| API 필드 | camelCase (DB snake_case ↔ Prisma 매핑) |
| 업무번호 | `CTR-YYMMDD-###`, `ORD-YYMMDD-###` (일 단위 시퀀스, 서버 생성) |
| 프론트 파일 | 컴포넌트 PascalCase.tsx, 그 외 kebab-case.ts |
| 커밋 | `feat|fix|refactor|docs|chore(scope): 요약` |

## 4. 테스트 전략

- 단위 테스트(Jest): 판정 로직(미주문/재출력/지연), 상태 전이 검증, 계약 확정 품목 펼침, 렌탈 가용 계산 등 **규칙 함수** 우선.
- 통합 테스트: 핵심 트랜잭션 API(계약 확정, 옵션 확정, 작업지시서 출력, 렌탈 배정 동시성)를 테스트 DB로 검증.
- 프론트는 MVP에서 수동 인수 시나리오(UI-01~16) 체크리스트로 갈음.

## 5. 로깅·운영

- 요청 로그: requestId(X-Request-Id 수신 또는 서버 생성), 사용자, 경로, 소요시간.
- 오류 로그와 감사로그 분리. 외부 연동 호출은 요청·응답 요약 별도 기록.
- 백업: `pg_dump` 일 1회 + storage 증분 복사(운영 스크립트는 Phase 7).
