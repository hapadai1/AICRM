## 1. 문제 정의

현재 `/payments`는 **목록 화면이 아니라 계약 1건의 결제 상세 화면**이다.

| 문제 | 근거 |
|---|---|
| 진입 시 아무 정보도 없음 | 계약을 선택하기 전까지 `Empty` 한 장. "이번 달 수금 내역"을 볼 수단이 없다 |
| 검색 기준이 계약(코드) 뿐 | 실무 진입점은 **고객**과 **날짜**인데 계약 셀렉트만 제공 |
| 전화번호 검색이 동작하지 않음 | placeholder에는 "전화번호"가 있으나 `ContractsService.list`의 where는 `contractNo`, `customer.name`만 본다 |
| 결제를 가로로 조회하는 API 부재 | 백엔드에 `GET /contracts/:id/payments` 밖에 없다 |

## 2. 개편 방향 — 2계층 분리

```
/payments                       → ① 결제 목록 (기본 화면, 신규)
/payments?contractId=<uuid>     → ② 계약 결제 상세 (기존 화면을 패널로 강등)
```

라우트를 새로 만들지 않고 **쿼리 파라미터로 전환**한다. 라우팅·권한 가드를 건드리지 않고, 뒤로가기로 목록에 복귀한다.

### 2.1 ① 결제 목록

- **검색조건**: 결제일 범위(기본 최근 1개월) · 고객(이름·전화) 또는 계약번호 통합 검색 `q` · 결제유형 · 상태 · 결제수단
- **고객 검색 팝업**(`shared/CustomerPickerModal`): 동명이인·비슷한 번호를 눈으로 확인하고 고객을 **정확히 특정**하는 경로. 선택하면 `q` 대신 `customerId`로 단건 조회하고, 검색창 자리에 고객 태그(이름·전화)를 표시한다. 태그 X로 해제, "변경"으로 재선택. 팝업 표에 상태·최근 방문일·계약 건수·**미수금**을 함께 보여 어느 고객인지 판별 가능하게 한다. 계약 목록 개편(06)에서도 재사용한다
- **결과 컬럼**: `결제일 · 고객명(전화) · 계약번호 · 계약구분 · 유형 · 금액 · 수단 · 상태 · 메모`
  - 정렬 기본값 `결제일 desc, createdAt desc`
- **상단 합계**: 수금 합계 / 환불 합계 / 순수금 / 건수 — 현재 필터 전체 기준(페이지 기준 아님)
- 행 클릭 → `?contractId=` 로 상세 진입
- 페이지네이션 표준(`page` 1-base, `size` 기본 30)

### 2.2 ② 계약 결제 상세

기존 요약 카드 + 결제 등록/취소/잔금 예정일 수정 기능을 **그대로 유지**하고 컴포넌트만 분리한다. 상단에 "목록으로" 이동 버튼을 둔다.

## 3. 백엔드 변경

### 3.1 신설 `GET /payments` (권한 `PAYMENT_VIEW`)

쿼리 파라미터 (`PaymentListQueryDto extends PageQueryDto`)

| 이름 | 타입 | 설명 |
|---|---|---|
| `dateFrom` / `dateTo` | `YYYY-MM-DD` | 결제일 범위(양끝 포함) |
| `q` | string | 고객명 · 고객 전화(하이픈 무시) · 계약번호 통합 검색 |
| `customerId` | uuid | 고객 단건 지정(고객 상세에서 진입) |
| `contractId` | uuid | 계약 단건 지정 |
| `paymentType` | enum | `DEPOSIT\|INTERIM\|BALANCE\|REPAIR_FEE\|REFUND\|ETC` |
| `status` | enum | `COMPLETED\|CANCELLED` |
| `paymentMethod` | string | 결제수단 |

응답 — 목록 envelope(`data` + `page`)에 `totals`를 덧붙인다.

```jsonc
{
  "data": [{
    "id": "...", "contractId": "...", "contractNo": "CTR-260721-001",
    "contractTypeName": "맞춤정장", "customerId": "...", "customerName": "홍길동",
    "customerPhone": "010-1234-5678",
    "paymentType": "DEPOSIT", "amount": 300000, "paymentDate": "2026-07-01",
    "paymentMethod": "카드", "status": "COMPLETED", "memo": "입금자: 홍길동",
    "createdAt": "..."
  }],
  "page": { "number": 1, "size": 30, "totalElements": 12, "totalPages": 1 },
  "totals": { "count": 12, "paidAmount": 3200000, "refundAmount": 100000, "netAmount": 3100000 }
}
```

- `totals`는 **현재 필터 전체** 기준이며 `status='COMPLETED'`만 집계한다. `REFUND` 유형은 `refundAmount`로 분리하고 `netAmount = paidAmount - refundAmount`.
- 전화번호 검색은 `customers.phoneNormalized`(하이픈 제거 저장)에 대해 입력값의 숫자만 남겨 `contains` 매칭한다.

### 3.2 계약 검색 보정

`ContractsService.list`의 `q` OR 절에 `customer.phoneNormalized contains` 추가 — 화면 placeholder와 실제 동작을 일치시킨다.

### 3.3 변경하지 않는 것

`GET /contracts/:id/payments`, `POST /contracts/:id/payments`, `POST /payments/:id/cancel`, `PATCH /contracts/:id/payment-schedule`은 계약 문서 04 §4 그대로 유지한다. 상세 패널이 계속 사용한다.

## 4. 프론트엔드 변경

| 파일 | 변경 |
|---|---|
| `api/payments.ts` | `PaymentListRow` · `PaymentTotals` · `searchPayments(params)` 추가 |
| `features/payments/PaymentsPage.tsx` | 목록 화면으로 교체(필터 바 + 합계 + 표), `contractId` 있으면 상세 패널 렌더 |
| `features/payments/ContractPaymentPanel.tsx` | 기존 요약·등록·취소·예정일 수정 로직 이전(신규 파일) |
| `shared/CustomerPickerModal.tsx` | 고객 검색 팝업(공용) — 이름·전화 검색, 미계약 포함 토글, 행 클릭 즉시 선택+조회 |

- Query Key: 목록 `['payments', 'list', params]`, 상세 `['contracts', contractId, 'payments']`(기존 유지)
- 등록·취소 성공 시 `['payments']`와 `['dashboard']`를 함께 invalidate
- 상태 배지는 텍스트+색 병기(구현표준 §2)

## 5. 테스트

`test/backend/payments.spec.ts`에 추가:

1. 날짜 범위 필터가 경계일을 포함한다
2. `q`로 고객명·전화번호(하이픈 유무 무관)·계약번호가 모두 검색된다

5. 페이지네이션 `page`/`size`
