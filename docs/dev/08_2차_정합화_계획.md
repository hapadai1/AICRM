## 1. 결론

`03_개발체크리스트.md`의 **"실서버 연동 정합화 ✅"는 사실이 아니다.** 실제로 정합화가 끝난 화면은 수선(방금 수정)·결제·고객 목록 정도이며, 나머지는 **삭제된 MSW mock의 응답 계약 위에 그대로 남아 있다.**

확인된 결함 **총 68건** — 화면이 죽는 크래시 **14건**, 값이 비거나 틀리게 보이는 것 **31건**, 요청이 거부되거나 조용히 버려지는 것 **23건**.

### 1.1 근본 원인 3가지

| # | 원인 | 결과 |
|---|---|---|
| A | **백엔드가 Prisma raw row를 그대로 반환**한다. 뷰(직렬화) 계층이 `appointment-view.ts` 한 곳에만 있다 | 중첩(`order.contract.customer`) · Decimal이 문자열(`"750000"`) · `@db.Date`가 ISO 일시(`2026-07-20T00:00:00.000Z`)로 그대로 노출 |
| B | **코드 체계의 단일 출처가 없다.** 상태 enum이 백엔드 상수 / 프론트 유니온 / 시드 데이터 3곳에 따로 산다 | 수선의 `COMPLETED`, 제작의 `RESERVED`처럼 **프론트 맵에 없는 코드 → `META[code].label` 크래시** |
| C | **`ValidationPipe({whitelist:true})`에 `forbidNonWhitelisted`가 없다** (`backend/src/main.ts:16`) | 잘못된 필드명이 **오류 없이 조용히 삭제**된다. 가봉 저장·계약 품목 금액·예약 고객 연결 등이 "성공"으로 표시되고 데이터만 유실 |

원인 C가 특히 위험하다 — P3 23건 중 대부분이 이것이고, **개발 중에 아무 신호도 주지 않는다.**

## 2. 해결 방향

### 2.1 응답 불일치 (원인 A) — 프론트 매퍼로 흡수

백엔드 응답을 정본으로 두고, **`api/*.ts`에서 화면용 뷰로 변환**한다. 방금 `api/repairs.ts`에 적용한 방식과 동일하다.

- 백엔드·백엔드 테스트(179건)를 건드리지 않아 회귀 위험이 낮다
- 변환 규칙 3종을 공용 유틸로 뽑는다: `toDateOnly(ISO)` · `toNumber(Decimal문자열)` · 중첩 평탄화
- 단, **백엔드에 데이터 자체가 없는 항목**은 매퍼로 해결 불가 → §4에서 별도 판단

### 2.2 코드 체계 (원인 B) — 안전 조회 강제

- 공용 헬퍼 [`frontend/src/shared/status-meta.ts`](../../frontend/src/shared/status-meta.ts) **신설 완료**. `metaOf(MAP, code)`는 미등록 코드를 코드값 그대로 표시한다
- `META[code].label` 직접 인덱싱 **35곳을 전수 교체**한다 → 코드가 어긋나도 화면이 죽지 않고 "낯선 코드"로 드러난다
- 프론트 상태 유니온을 백엔드 상수 기준으로 재정의한다(수선에서 이미 수행)

### 2.3 조용한 필드 유실 (원인 C) — 파이프 강화

`backend/src/main.ts`의 전역 파이프에 **`forbidNonWhitelisted: true`** 를 켠다. 알 수 없는 필드는 400으로 즉시 실패한다.

> 켜는 순간 §5의 P3 요청 불일치가 전부 400으로 드러난다. **P3 수정을 먼저 끝낸 뒤 마지막에 켠다.**

## 4. P2 (값이 비거나 틀림, 31건) — 판단이 필요한 항목

대부분은 필드명 매핑으로 끝나지만, **아래는 백엔드에 데이터가 아예 없다.** 화면에서 제거할지, 백엔드를 확장할지 결정이 필요하다.

| 항목 | 화면 | 제안 |
|---|---|---|
| `order.timeline` 상태 타임라인 | ORD-001 | 감사로그로 대체 가능 → **화면에서 제거** 권장 |
| `item.optionProgress` 옵션 진행률 | ORD-001 | 옵션 세션에서 집계 가능 → **백엔드 확장** 권장 |
| `workOrder.printable` 출력 가능 판정 | WO-002 | 현재 **Excel 출력이 영구 비활성** → **백엔드 확장 필수** |
| `customer.lastTransactionType` 최근 거래 유형 | CUST-001 | 필터는 동작하는데 열만 빈칸 → **백엔드 enrich에 추가** |
| `customer.inactiveReason` 비활성 사유 | CUST-002 | DB 컬럼 없음(감사로그만) → **화면에서 제거** |
| `task.acknowledgedBy` 확인 처리자 | DASH-001 | DB 컬럼 없음 → **제거 또는 스키마 추가** |
| `payment.cancelReason` 취소 사유 | PAY-001 | DB 컬럼 없음(감사로그만) → **제거 또는 스키마 추가** |
| `permission.domain/group` 권한 분류 | ADMIN-003 | 권한 체크표가 1행으로 붕괴 → **백엔드 확장 필요** |

그 외 P2는 전부 이름/형태 매핑이다: 계약 목록·상세 전 필드, 옵션 세션 전 필드, 렌탈 구성품·출고일, 알림 발송 이력, 사용자·역할·감사로그, ADMIN-002 옵션 화면(`category`↔`productCategory`로 **통째로 빈 화면**).

## 5. P3 (요청 거부·유실, 23건) — 기능이 아예 동작하지 않음

| 기능 | 증상 |
|---|---|
| 계약 확정 / 변경확정 / 취소 | `version`↔`rowVersion` 불일치로 **항상 실패** |
| 계약 품목 금액·비고 저장 | `amount`/`note`가 whitelist에 삭제 → **금액 0으로 저장** |
| 계약 구분 추가 / 복제 | `code` 미전송 → 항상 400 |
| 채촌 신규 생성 / 수정 / 복사 | 필드명 전부 불일치 → **신규 채촌 생성 불가** |
| 제작 상태 변경 / 제작 요청 | `toStatus`↔`newStatus` → 항상 400 |
| 제작 입·출고일 | `receivedDate`↔`receivedAt` → **고른 날짜 무시, 항상 오늘** |
| 가봉 저장 | 대상 구성품·보정내용·다음방문일 **전량 유실**(201 성공으로 표시) |
| 예약 고객 연결 / 이름·전화 수정 | DTO에 필드 없음 → 실패 또는 무반영 |
| 렌탈 ID변경 후보 조회 / 가용 조회 | `componentType` 필수인데 미전송 → 400 |
| 사용자 추가 / 역할 권한 저장 | 필드명 불일치 + 비밀번호 입력란 자체가 없음 → 400 |
| 옵션 세트 버전 복사 / 단계 저장 | 필드명 불일치 → 빈 버전 생성 또는 400 |
| 대시보드 공유메모 완료 토글 | `completed`↔`status` → **영구히 동작 안 함** |
| 감사로그 대상 검색 / 기본 기간 | `query` 무시, `to`가 00:00Z라 **오늘 로그가 항상 누락** |
| 알림 수신번호 / SMS 대체발송 | 조용히 삭제 → 입력한 번호 무시 |

## 7. 재발 방지

1. **`forbidNonWhitelisted`** — 요청 측 불일치를 개발 시점에 400으로 노출
2. **`metaOf` 강제** — 응답 측 코드 불일치가 크래시 대신 표시로 격하
3. **시드 데이터 검증** — 시드가 백엔드 상수에 없는 코드를 쓰지 못하게 한다(수선 `PRODUCTION`·`COMPLETED`가 실제로 그랬다)
4. **화면 스모크 테스트** — 전 화면 진입 + 주요 목록 렌더까지 확인하는 Playwright 시나리오. 이번 결함 14건은 **화면을 한 번 열어보기만 해도** 전부 잡혔다
