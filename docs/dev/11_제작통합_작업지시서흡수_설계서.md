## 1. 현행 진단 (왜 통합인가)

두 기능은 같은 `OrderItem`(품목)의 제작 여정을 다루면서 **FK 하나 없이 완전히 분리**돼 있다. 공통 조상은 `OrderItem` 뿐이다.

| | 작업지시서 (Work Order) | 제작·입출고 (Production) |
|---|---|---|
| 정체 | 옵션 확정 + 채촌 스냅샷 **Excel 출력물**(버전) | 발주 이후 **실물 상태 추적** |
| 앵커 | `WorkOrder.orderItemId @unique` (품목 1:1) | `OrderItem.status` + `OrderItemComponent.status` + `ProductionEvent` 로그 |
| 상태 | 미주문/재출력필요/최신/준비미완 — **저장 안 함, 읽을 때 계산** | 제작요청→제작중→가봉→제작완료→입고→출고 — **저장** |
| 상태 근거 | `resolveWorkOrderStatus()` = 옵션 세션 confirmedAt + 채촌 link linkedAt + 최신버전 issuedAt ([work-orders.service.ts:68](../../backend/src/modules/work-orders/work-orders.service.ts#L68)) | `ITEM_STATUS_FLOW` / `computeAggregateStatus` ([production-status.ts](../../backend/src/modules/production/production-status.ts)) |
| Excel | 저장·버전관리 (`work-orders/{versionId}.xlsx` + `File`) | 가봉지 즉석 생성(미저장) |
| 상호 FK | **없음** | **없음** |

**단절의 증거**: 제작·입출고의 "제작 요청" 확인창이 *"작업지시서 출력 여부는 이 목록에서 확인할 수 없습니다"* 라고 경고한다 ([ContractProductionPage.tsx:138](../../frontend/src/features/production/ContractProductionPage.tsx#L138)). 통합하면 이 경고 자체가 사라진다.

**용어 정리**: 작업지시서 상태는 실제 라이프사이클이 아니라 "**출력 게이트**"다 — *현재 사양이 출력됐나 / 출력본이 낡았나*. 제작·입출고 상태가 실제 물리 진행이다. 통합 화면에서 둘을 한 행에 나란히 보이게 하면 개념이 명확해진다.

## 2. To-Be 통합 구조

제작·입출고 화면(계약 단위)이 전 라이프사이클을 관장하는 **제작 코크핏**이 된다.

```
계약 (1건)  ──  ProductionPage(/production)에서 계약별 롤업
  └ 품목별 행:
       [작업지시서 상태·출력 Excel] → [제작요청] → [가봉] → 구성품[입고] → 구성품[출고]
         └ 흡수 대상(신규)              └────────── 이미 제작·입출고에 존재 ──────────┘
```

- 관리·묶음 단위 = **계약**. Excel 문서 단위 = **품목**(변경 없음).
- 작업지시서 **메뉴 제거**, 목록 화면 제거. 미리보기/출력 화면(`WorkOrderPreviewPage`)은 **유지**하되 진입 동선만 코크핏에서 연다.

---

## 3. 데이터 모델 — 변경 없음

`관리 단위만 1:1` 결정에 따라 **스키마 변경·마이그레이션 없음**.

- `WorkOrder`(`orderItemId @unique`), `WorkOrderVersion`, `ProductionEvent`, `OrderItemComponent`, `FittingSession` 모두 그대로.
- 계약 연결은 기존과 동일하게 간접(`OrderItem → Order.contractId`).
- 이미 반영된 것: 작업지시서 목록 행에 `contractId`/`contractNo` 추가([work-orders.service.ts:572](../../backend/src/modules/work-orders/work-orders.service.ts#L572)). 본 통합에서 작업지시서 목록 화면은 폐기되므로(§7) 이 필드는 제작 응답 확장(§5)으로 대체 활용한다.

## 4. 화면 설계

### 4.1 계약 롤업 목록 — `ProductionPage` (`/production`)

현행: 계약별 그룹 + 품목 집계 태그(전체 출고/전체 입고 등) + [열기]. **유지**하되 작업지시서 집계를 한 열 추가한다.

| 열 | 내용 | 변경 |
|---|---|---|
| 계약 / 고객 | contractNo + customerName | 유지 |
| 맞춤 품목 | 건수 + 제작 집계 태그 | 유지 |
| **작업지시서** | **미출력 n · 재출력필요 n** 요약 태그 | **신규** |
| 액션 | [열기] → `/contracts/:id/production` | 유지 |

### 4.2 제작 코크핏 — `ContractProductionPage` (`/contracts/:id/production`)

현행 품목 테이블(품목·주문·집계상태·구성품 진행·액션) + 구성품 확장(입고/출고)을 유지하고, **작업지시서 열과 출력 액션을 통합**한다.

품목 행 (좌 → 우로 라이프사이클 순서):

| 열 | 내용 | 출처 |
|---|---|---|
| 품목 / 주문 | displayName + orderNo | 기존 |
| **작업지시서** | 상태 뱃지(준비미완/미주문/재출력필요/최신) + **[출력 Excel]** 버튼 | **흡수** — 상태 = `resolveWorkOrderStatus`, 버튼 → `/work-orders/:orderItemId` |
| 제작 상태 | 품목 집계 뱃지 | 기존 |
| 구성품 진행 | 구성품별 태그 | 기존 |
| 액션 | [제작요청] · [가봉 기록] | 기존 |

- **[출력 Excel]** 은 새 API가 아니라 **기존 미리보기/출력 화면**(`WorkOrderPreviewPage`, `POST /order-items/:id/work-order-versions`)으로 진입한다. 출력 로직·버전·Excel 저장은 무변경.
- 작업지시서 상태가 `준비미완`(옵션 미확정 or 채촌 미완)이면 [출력] 비활성 + 툴팁(기존 규칙 재사용).
- "제작 요청" 확인창의 *"작업지시서 출력 여부 확인 불가"* 경고 문구 **삭제** — 이제 같은 행에서 보인다.

구성품 확장 행(입고/출고): **현행 그대로**.

### 4.3 작업지시서 미리보기/출력 — `WorkOrderPreviewPage` (`/work-orders/:orderItemId`)

**유지.** 진입 동선만 코크핏의 [출력] 버튼으로 일원화. 출력 후 돌아오면 코크핏 상태가 `최신`으로 갱신(react-query invalidate).

### 4.4 메뉴 개편 — `AppLayout.tsx`

맞춤 제작 그룹에서 **작업지시서 제거**:

```
맞춤 제작
 ├ 스타일 컨설팅   (/options)          ← 유지
 └ 제작·입출고     (/production)        ← 작업지시서 흡수, 코크핏
```

- `작업지시서`(`/work-orders`) 메뉴 항목 삭제.
- (검토) 제작·입출고를 `제작 관리`로 리네임할지는 §9 미결.

## 5. API 설계 (변경 최소)

**핵심: 코크핏이 한 번의 호출로 작업지시서 상태까지 받도록 `GET /production/items` 응답을 확장한다.** 별도 작업지시서 호출·클라이언트 조인을 피한다.

### 5.1 `GET /production/items` 응답에 `workOrder` 서브객체 추가

`production.service.listProductionItems`의 include에 작업지시서 판정 소스를 추가하고(현재 `orderItemInclude`가 work-orders 모듈에 있으므로 공유 헬퍼로 추출 또는 동일 include 재현), 품목별로 다음을 계산해 내려준다:

```ts
workOrder: {
  workOrderId: string | null;
  status: 'WAITING' | 'UNORDERED' | 'REPRINT_NEEDED' | 'CURRENT';  // resolveWorkOrderStatus 재사용
  currentVersionNo: number | null;
  lastIssuedAt: string | null;   // YYYY-MM-DD HH:mm
  canIssue: boolean;             // status !== 'WAITING'
}
```

- `resolveWorkOrderStatus()`와 판정 include를 **work-orders 모듈에서 공유 유틸로 추출**해 production 서비스가 재사용(중복 판정 로직 금지 — 표준 §단일 출처).
- 권장 이유: 코크핏 렌더가 1콜로 끝나고, 작업지시서 목록 API(`GET /work-orders`)는 폐기 가능해진다.

### 5.2 무변경 유지

- 출력: `POST /order-items/:id/work-order-versions`, 미리보기: `GET /order-items/:id/work-order/preview`, 다운로드: `GET /work-order-versions/:id/file` — 전부 유지.
- 제작·가봉·입출고 엔드포인트 전부 유지.

## 6. 상태·라이프사이클 통합 뷰

한 행에서 **출력 게이트(작업지시서) → 물리 진행(제작·입출고)** 이 순서대로 보인다.

```
작업지시서:  준비미완 ─→ 미주문 ─→ (출력) ─→ 최신 ⇄ 재출력필요(옵션·채촌 변경 시)
제작·입출고:                           제작요청 → 제작중 → 가봉 → 제작완료 → 입고 → 출고
```

- 두 상태는 **독립 저장 그대로**(작업지시서=계산, 제작=저장). 화면에서만 나란히 표시.
- **커플링 없음(확정)**: "Excel 출력 = 제작요청"으로 잇는 로직·전제·경고를 두지 않는다. `[제작요청 완료]`는 작업지시서 상태와 무관하게 눌리는 독립 버튼. 담당자가 두 상태를 보고 판단한다.

## 7. 폐기·정리 대상

| 대상 | 처리 |
|---|---|
| `작업지시서` 메뉴 항목 (`AppLayout.tsx:52`) | 삭제 |
| `WorkOrderListPage.tsx` (직전 작업의 계약그룹 목록) | **삭제** — `ProductionPage` 롤업으로 대체 |
| 라우트 `/work-orders` (목록) | 삭제 또는 `/production`으로 redirect |
| 라우트 `/work-orders/:orderItemId` (미리보기) | **유지** |
| `GET /work-orders` (목록 API) | 폐기 후보 (코크핏이 §5.1로 대체). 단계적으로 제거 |
| `WorkOrderListRow.contractId/No` (직전 추가) | 목록 폐기 시 함께 제거, 판정 로직은 §5.1 공유 유틸로 이관 |

> 주의: 직전 작업에서 만든 **작업지시서 계약그룹 목록 화면은 본 통합으로 대체·폐기**된다. 백엔드에 넣은 `contractId/contractNo`와 `resolveWorkOrderStatus`는 §5.1 공유 유틸로 재활용하므로 헛작업은 아니다.

## 8. 구현 순서 (승인 후)

- **Phase A — 백엔드**: `resolveWorkOrderStatus` + 판정 include를 공유 유틸로 추출 → `GET /production/items` 응답에 `workOrder` 서브객체 추가(§5.1). 기존 work-orders 테스트 유지, production 테스트에 workOrder 필드 검증 추가.
- **Phase B — 프론트 코크핏**: `ContractProductionPage`에 작업지시서 열 + [출력] 버튼 통합, "제작요청" 경고 문구 제거, 출력 후 invalidate. `production.ts` API 타입에 `workOrder` 추가.
- **Phase C — 롤업·메뉴**: `ProductionPage`에 작업지시서 집계 열 추가. `AppLayout`에서 작업지시서 메뉴 제거. `/work-orders` 목록 라우트 redirect, `WorkOrderListPage` 삭제.
- **Phase D — 정리**: `GET /work-orders` 목록 API·`WorkOrderListRow` 잔여 필드 제거(회귀 확인 후).

각 Phase는 `tsc --noEmit` 양측 통과 + 관련 spec 통과를 완료 기준으로 한다.

## 9. 미결 결정 (확정)

1. **메뉴명**: `제작·입출고` → **`제작 관리`로 리네임** (작업지시서까지 포함하므로).
2. **롤업 집계 열**: `ProductionPage`에 작업지시서 요약(미출력·재출력필요 건수) 열 **추가**.
3. **제작요청 게이트**: **커플링 없음.** "Excel 출력 = 제작요청" 로직/전제 두지 않음. 제작요청은 독립 버튼 **`[제작요청 완료]`** 로 유지하고, 작업지시서 관련 경고 문구 제거. (§6의 "미출력 시 경고" 제안 폐기.)

## 10. 영향·리스크

- **데이터 모델 무변경** → 마이그레이션 없음, 회귀 위험 낮음.
- 출력/버전/Excel 저장 로직 **무변경** → 작업지시서 핵심 기능 회귀 위험 없음.
- 주 위험은 **UI 병합**(코크핏 열 증가)과 **판정 로직 공유 이관**(중복 제거) 두 곳. spec으로 커버.
- 권한: 코크핏에서 출력 진입 시 기존 작업지시서 출력 권한 체크 유지. 제작 액션은 `PRODUCTION_EDIT` 유지.
