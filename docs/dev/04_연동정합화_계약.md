# 화면↔백엔드 연동 정합화 계약 v1.0 (2026-07-21)

전수 조사에서 발견된 차이의 **확정 해소 방안**. 프론트·백엔드 수정 작업은 모두 이 문서를 기준으로 한다.
원칙: 요청 필드명 = 백엔드(설계서) 기준으로 프론트 수정 / 응답 구조 = 화면 요구대로 백엔드에 뷰 매퍼 추가 / 화면 전용 목록 API = 백엔드 신설.
mock 핸들러도 프론트 수정 시 **아래 확정 계약과 동일한 형태**로 갱신한다(실서버·mock 동일 계약 유지).

## 0. 스키마 (적용 완료)
- `measurement_sessions.completed_at timestamptz NULL` — 채촌 완료 마커(감사로그 방식 폐기)
- `contracts.balance_due_date date NULL` — 잔금 결제 예정일(결제 지연 판정 기준)

## 1. 예약 (APPT) — 백엔드 수정
- 신설: `POST /appointments/:id/confirm`, `POST /appointments/:id/no-show`, `POST /appointments/:id/resolve-conflict { resolution: 'NAVER'|'CRM' }`, `GET /appointment-purposes`(APPOINTMENT_VIEW, active 목록)
- 목록 쿼리: `purposeCodes`(콤마 목록)·`statuses`(콤마 목록) 지원(기존 단수 파라미터도 유지)
- 응답 뷰 매퍼(목록·상세·액션 공통 평면 필드): `startAt`(=scheduledStart), `endAt`, `memo`(=notes), `version`(=rowVersion), `customerId/customerName/phone/customerStatus`, `purposeCode/purposeName`, `source`, `status`, `syncStatus`(CRM이면 NORMAL, NAVER는 localOverride→LOCAL_EDITED, naverUpdatedAt>syncedAt→NAVER_CHANGED, 둘 다→CONFLICT), `naverReservationId`(=externalId). 상세에는 `consultations[]` 포함
- 상담: `POST .../consultations`가 `{ interests?: string[], content }` 수용(interests는 consultation_category에 콤마 저장). 응답 뷰: `{ id, interests[], content, createdBy(표시명), createdAt }`
- 프론트 수정: 생성/수정 요청을 `scheduledStart/scheduledEnd/notes`로 전송(응답은 뷰 필드 그대로 사용)

## 2. 고객 (CUST) — 백엔드 수정
- `GET /customers/:id` 응답 재구성: `{ customer:{...평면}, summary:{ contractCount, totalAmount, paidAmount, balanceAmount }, appointments[], consultations[], contracts[](뷰: id, contractNo, contractTypeName, status, currentVersionNo, totalAmount, depositAmount, balanceAmount, contractedAt), orders[], measurements[](세션 요약), components[], rentals[](배정 요약), repairs[], payments[] }`
- `GET /customers` 쿼리: `status`(기존) + `includeProspect=true`(→PROSPECT 포함) + `transactionType`(CUSTOM/RENTAL — 주문 보유 고객 필터) 지원
- 프론트: 목록 토글은 `includeProspect=true` 전송. 고객 검색 공통 파라미터는 `q` + 필요 시 `status=ALL`(notifications의 `query`, measurements 검색 수정)

## 3. 계약 (CONT)
- 백엔드: 목록 쿼리 `search` 유지 + `q` 별칭·`status` 필터 추가. `POST /contracts/:id/revisions` body `changeReason`(프론트가 이름 변경) + 확정 `POST .../revisions/:rid/confirm`에 `changeReason?, totalAmount?, depositAmount?, lines?` 수용(확정 직전 revision에 반영 후 확정)
- 프론트: `reason`→`changeReason` 전송
- **목록 확장** (개편계획 06): `GET /contracts` 쿼리에 `dateField(contractedAt|paymentDate|completionDueDate, 기본 contractedAt)`·`dateFrom/dateTo`(양끝 포함)·`contractTypeId`·`unpaidOnly`·`sort(필드,asc|desc)` 추가. `q` OR 절에 고객 전화(`phoneNormalized`, 숫자만 비교)·계약 구분명 추가. 각 행에 `paidAmount`(완료 결제 합계 − 환불)·`unpaidAmount`(계약금액 − 수납액, 음수는 과납)·`lastPaymentDate` 평면 필드 추가, envelope 최상위에 `totals { count, totalAmount, paidAmount, unpaidAmount }`(필터 전체 기준). 기본 정렬은 계약일 내림차순(계약일 없으면 등록일)
- 수납액 집계는 Prisma where로 표현 불가 → 필터 결과를 모두 읽어 결제 groupBy를 병합한 뒤 메모리 정렬·페이징한다(단일 매장 데이터량 전제). 계획서 06 §3.2의 `$queryRaw` 안 대비 단순화한 구현이며 응답 계약은 동일하다

## 4. 결제 (PAY)
- 백엔드: 결제 유형 `DEPOSIT | INTERIM | BALANCE | REPAIR_FEE | REFUND | ETC` (기존 OTHER는 ETC로 통합·하위호환 수용). `payerName?` 수용(memo에 `입금자: {payerName}` 병합). 목록 응답 summary 확장: `{ contractNo, customerName, contractTypeName, contractAmount, paidAmount, balanceAmount, balanceDueDate }`. 신설 `PATCH /contracts/:id/payment-schedule { balanceDueDate }`(PAYMENT_EDIT). 대시보드 PAYMENT_DELAY 판정을 balanceDueDate 기준으로 변경(없으면 판정 제외)
- 프론트: 요청 필드 `paymentDate/paymentMethod` 로 변경, 응답 `{ payments, summary, warning }` 구조에 맞춤(초과 여부 = `warning?.code==='OVER_COLLECTION'`), 뷰 필드 `paymentDate/paymentMethod` 사용
- **결제 목록 신설** (개편계획 07): `GET /payments`(PAYMENT_VIEW) — 쿼리 `dateFrom/dateTo/q/customerId/contractId/paymentType/status/paymentMethod` + 페이지네이션. 행에 `contractNo/contractTypeName/customerId/customerName/customerPhone` 포함, 목록 envelope 최상위에 `totals { count, paidAmount, refundAmount, netAmount }`(필터 전체 기준·COMPLETED만·REFUND 분리). `q`는 계약번호·고객명·고객 전화(숫자만 비교)를 함께 검색
- 결제 상태 표기 정정: 프론트 `Payment.status`는 `COMPLETED | CANCELLED` (기존 `PAID` 오타로 취소 버튼이 노출되지 않던 문제 해소)
- 목록 envelope 공통: `Paginated`에 `extra`를 추가해 `data/page` 옆에 부가 필드를 실을 수 있게 함(프론트 인터셉터는 `meta` 제외 전 필드를 그대로 전달)
- 계약 검색(`GET /contracts?q=`) OR 절에 `customer.phoneNormalized` 추가 — placeholder와 실제 동작 일치

## 5. 렌탈 (RENT)
- 백엔드 신설: `GET /rental-allocations?view=pickup|return&date=`(RENTAL_VIEW — pickup: RESERVED/PREPARING & pickupDate<=date, return: CHECKED_OUT, 지연 포함. 뷰: allocation + 실물 managementCode + 고객/주문), `GET /rental-orders/components?orderId?`(렌탈 주문 구성품 + 현재 배정)
- 백엔드 수용 확장: checkout `confirmedItemCode`(관리코드) 또는 `confirmedInventoryItemId` 둘 다 허용, allocation 생성에 `itemCode` 대체 허용(코드→id 해석)
- 프론트: `returnedDate`→`returnDate`, 재고 등록 `managementCode`(필수)/`componentType`/`notes`, import는 `{ dryRun?, items:[{managementCode?, componentType, design, color, size, quantity?}] }` 형태, 목록·가용 필터 `componentType/skuSize/availableOn/size`, PATCH `notes`, 상태 변경 `newStatus`, 응답 필드 `managementCode/componentType/notes` 사용(RentalItem 매핑 수정)

## 6. 옵션 (OPT) — 백엔드 신설
- `GET /order-items/option-progress`(OPTION_SELECT): 맞춤 품목별 `{ orderItemId, displayName, productCategory, customerName, orderNo, fabric, status, completedStages, totalStages, sessionId? }`
- `GET /order-items/:id/option-session`(OPTION_SELECT): 품목의 현재(is_current) 세션 상세(단계·선택지·선택값·재개 지점). 세션 없으면 `{ session: null }`
- `POST /order-items/:id/option-sessions` body `{ fabric? }` → fabricName 저장

## 7. 제작 (PROD) — 프론트 수정
- 목록 경로 `/production/order-items` → `/production/items` 로 변경(응답 형태는 백엔드 기존 유지, 프론트 매핑 확인)

## 8. 수선 (REPAIR) — 백엔드 신설
- `GET /repairs/link-targets?customerId=`(REPAIR_VIEW, **`:id` 라우트보다 먼저 선언**): `{ orderItems[](맞춤, 구성품 포함), rentalItems[] }`

## 9. 채촌 (MEAS) — 백엔드 리팩터링
- `completed_at` 컬럼 사용으로 전환: complete 시 기록, 완료 판정·수정 차단·품목 연결 선행조건 모두 컬럼 기준(감사로그 마커 제거). 목록·상세 응답에 `completed: boolean`, `completedAt` 포함(기존 유지)

## 10. 대시보드 (DASH) — 백엔드 수정
- `GET /dashboard/summary` 응답: `{ date, appointments[](1번 예약 뷰 필드), week:[{date, count}](오늘±3일), taskCounts:{ LATE_RETURN, INBOUND_DELAY, PAYMENT_DELAY, UNORDERED, REPRINT_NEEDED } }`

## 11. 기준정보 (ADMIN) — 프론트 수정
- slug `appointment-purposes`(복수)로 통일. 백엔드 미지원 타입(품목·구성품/결제수단/수선 구분) 탭은 "백엔드 연동 예정" 안내로 비활성화(mock에서는 동작 유지 가능하되 실서버에서 오류 안 나게 가드)
