## 1. 확정 기술 스택

| 구분 | 확정안 | 비고 |
|---|---|---|
| 언어 | TypeScript (strict) | 프론트·백엔드 공통 |
| 프론트엔드 | React 18 + Vite | 설계서 권장안 채택 |
| UI 라이브러리 | Ant Design 5 | 표·폼 중심 CRM에 적합, 한국어 로케일 내장 |
| 서버 상태 | TanStack Query v5 | 목록/상세 캐싱, 낙관적 잠금 재조회 흐름 |
| 클라이언트 상태 | Zustand | 인증 토큰·전역 UI 상태 최소 사용 |
| 라우팅 | React Router v6 | 화면 ID 기반 경로 구성 |
| 백엔드 | NestJS 10 (REST) | 설계서 권장안 채택 |
| ORM | **Prisma** (TypeORM 대신 확정) | 스키마 선언·마이그레이션·타입 안전성 우위 |
| DB | PostgreSQL 16 | `btree_gist` 확장 필수(렌탈 기간 배타 제약) |
| 인증 | JWT access(30분) + refresh(14일, 서버 저장·회전) | 설계서 12.2 준수 |
| Excel | exceljs | 작업지시서 출력. 공장 양식 수령 후 셀 매핑 교체 |
| 파일 저장 | 로컬 볼륨(`storage/`) + files 테이블 메타 | 온프레미스 NAS 마운트로 교체 가능 |
| 배포 | 온프레미스 직접 실행 (Docker 미사용 확정) | Node 프로세스 + 로컬 PostgreSQL 16 |
| 테스트 | Jest + supertest, 루트 `test/` 폴더 통합 관리 | 전용 `aicrm_test` DB 사용 |

## 2. 프로젝트 구조 (모노레포)

```
AICRM/
├─ docs/                  # plan(설계), dev(개발 산출물)
├─ test/
│  └─ backend/            # 백엔드 테스트 전체 (Jest + supertest, aicrm_test DB)
├─ backend/
│  ├─ prisma/
│  │  ├─ schema.prisma    # 43개 테이블 (데이터모델설계서 16장 기준)
│  │  └─ seed.ts          # 권한·역할·관리자·기준정보·채촌항목·옵션세트 시드
│  └─ src/
│     ├─ common/          # 응답 포맷, 오류코드, 필터, 데코레이터, 감사 인터셉터
│     ├─ config/          # 환경변수 검증
│     ├─ prisma/          # PrismaService
│     └─ modules/
│        ├─ auth/  users/  audit/  files/          # Phase 1
│        ├─ customers/  appointments/  contracts/  # Phase 2
│        ├─ orders/  options/  measurements/       # Phase 3~4
│        ├─ work-orders/  production/              # Phase 4~5
│        ├─ rentals/  repairs/  payments/          # Phase 5~6
│        └─ notifications/  dashboard/             # Phase 7
└─ frontend/
   └─ src/
      ├─ api/             # axios 클라이언트, 인터셉터(토큰 갱신·오류 변환)
      ├─ app/             # 라우터, 레이아웃(메뉴), 인증 가드
      ├─ features/<도메인>/ # 화면 단위 (화면 ID와 1:1 매핑)
      └─ shared/          # 공용 컴포넌트(상태 배지, 확인 다이얼로그, 키패드 등)
```

## 3. 개발 단계와 완료 기준 (통합설계서 20.1 매핑)

| Phase | 범위 | 완료 기준 |
|---|---|---|
| 1. 기반 | 스캐폴드, DB 스키마 전체, 인증/JWT, 사용자·역할·권한, 감사로그, 파일, 공통 규약(응답·오류·낙관적잠금) | 로그인→대시보드 진입, 권한 가드 동작, 감사로그 기록 |
| 2. 예약·고객·계약 | 예약 CRUD(+네이버 수집 어댑터 stub), 상담, 고객 상태 전환, 계약 구분, 계약서 작성·확정·변경 버전 | 인수 시나리오 UI-01~05 |
| 3. 주문·옵션 | 계약 확정 시 주문·품목·구성품 생성, 옵션 세트 버전 관리, 임시저장·재개·확인서·확정 | UI-06, UI-07 |
| 4. 채촌·작업지시서 | 채촌 세션·태블릿 키패드, 버전 비교, Excel 출력·버전·스냅샷, 미주문/재출력 판정 | UI-08~10 |
| 5. 제작·수선 | 제작 상태 이벤트, 구성품 부분 입출고, 가봉, 수선 접수·상태 | UI-11 |
| 6. 렌탈 | SKU·실물 등록(개별/CSV), 기간 가용 검색, ID 배정·변경·출고·반납, 배타 제약 | UI-12~15 |
| 7. 운영 고도화 | 대시보드 판정·확인 처리, 공유 메모, 결제, 알림톡/SMS 어댑터, 통계 | UI-16, API 인수 기준 전체 |

각 Phase 종료 시: 마이그레이션 반영, 핵심 서비스 단위 테스트, 해당 인수 시나리오 수동 확인.

## 4. 미확정 항목 구현 방침 (통합설계서 부록 C 대응)

| 항목 | MVP 구현 방침 |
|---|---|
| 네이버 예약 연동 | `NaverReservationAdapter` 인터페이스 + 수동 트리거 sync API. 실제 수집 로직은 방식 확정 후 구현체만 교체 |
| 계약서/작업지시서 양식 | 기본 템플릿으로 출력. 실제 양식 수령 시 셀 매핑 모듈만 교체 |
| 알림톡 벤더 | `MessageVendorAdapter` 인터페이스 + 콘솔/DB 기록 stub. 발송 이력 구조는 실제와 동일 |
| PG | payments에 `external_tx_id`, `raw_event(jsonb)` 컬럼만 확보, 수기 입력 |
| 역할 매트릭스 | 시드: SUPER_ADMIN(전체), MANAGER(관리자 메뉴 제외 전체), STAFF(조회+일반 업무) — 운영 화면에서 조정 |
| 옵션 단계 필수 여부 | 전 단계 `required=true` 기본값 |

## 5. 실행 환경

- Node.js 20 LTS, npm. Docker는 사용하지 않는다.
- DB: 로컬 PostgreSQL 16. 기존 DB를 재사용하지 않고 전용 DB를 새로 만든다 — 개발 `aicrm_dev`, 테스트 `aicrm_test` (role `aicrm`)
- 환경변수: `backend/.env` (`.env.example` 참조) — DATABASE_URL, JWT_SECRET, FILE_STORAGE_PATH 등
- 타임존: DB는 UTC(timestamptz), 표시 Asia/Seoul. 업무 날짜(date)는 그대로 저장
- 테스트: 루트 `test/backend`에 모든 테스트 소스 통합. `cd backend && npm test`
